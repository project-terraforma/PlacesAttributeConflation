# Replay Corpus v1 Operational Workflow

This workflow builds the first website-heavy PAC replay corpus using the merged replay, corpus-gate, dorking, priority, review, and evidence-enrichment tools.

The goal is not to collect random conflicts. The goal is to collect replayable website-centered evidence situations that stress stale websites, wrong official-looking pages, identity drift, branch confusion, aggregator echo, and abstention behavior.

## Corpus v1 target

Target shape:

- 650 total replay cases
- 425 website-heavy cases
- 100 identity-drift cases
- 60 category/taxonomy cases
- 40 source-dependency / aggregator-echo cases
- 25 abstention / weak-evidence / tie cases

Gate thresholds are implemented in `ReplayCorpusV1Thresholds`.

## Step 1: Generate prioritized first-pass workplan

Start from a conflict dork CSV generated by the existing conflict/dork pipeline.

```bash
python scripts/build_prioritized_workplan.py \
  --input reports/ranker/conflict_dorks.csv \
  --output-dir reports/workplans/pac_v1_first50 \
  --cases-per-batch 25 \
  --batch-count 2
```

Outputs:

- `reports/workplans/pac_v1_first50/batch_001.csv`
- `reports/workplans/pac_v1_first50/batch_002.csv`
- `reports/workplans/pac_v1_first50/evidence_template_001.csv`
- `reports/workplans/pac_v1_first50/evidence_template_002.csv`
- `reports/workplans/pac_v1_first50/manifest.json`

The workplan prioritizes P0 website-heavy cases first:

- missing website
- aggregator/social-as-website
- chain homepage / branch ambiguity
- Overture/GERS website gap
- identity-drift website signals
- baseline-wrong website predictions

## Step 2: Fill evidence URLs

Open each `evidence_template_*.csv` and add explicit evidence URLs for each case.

The fetcher does not perform search. This is intentional. Replay requires explicit evidence URLs, not hidden live search state.

Recommended evidence mix per hard case:

- official or official-like page
- contact/about/location page when available
- directory/social page when it is part of the conflict
- stale/old-domain page when relevant
- branch/location page for chain cases

## Step 3: Fetch and enrich evidence pages

Use extended schema for replay-grade website evidence.

```bash
python scripts/fetch_evidence_pages.py \
  --extend-schema \
  --full-text-dir reports/evidence/full_text \
  --excerpt-chars 10000 \
  reports/workplans/pac_v1_first50/evidence_template_001.csv \
  reports/workplans/pac_v1_first50/evidence_template_002.csv
```

Extended evidence columns include:

- `final_url`
- `redirected`
- `http_status`
- `content_hash`
- `canonical_url`
- `domain`
- `registered_domain`
- `page_text_full_path`
- `page_text_excerpt`
- `schema_org_detected`
- `localbusiness_schema_detected`
- `detected_phone`
- `detected_address`
- `detected_name`
- `detected_status`
- `identity_claims`
- `fetch_status`
- `fetch_error`

## Step 4: Seed or merge replay episodes

Use the existing replay seed / replay batch path for collected evidence.

Example shape:

```bash
python scripts/run_harness.py replay-seed \
  --batch reports/workplans/pac_v1_first50/batch_001.csv \
  --evidence reports/workplans/pac_v1_first50/evidence_template_001.csv
```

Then merge collected replay files into one corpus.

```bash
python scripts/run_harness.py replay-merge \
  --input-dir reports/replay_collected \
  --output reports/replay/merged_v1_first50.json
```

If the current harness command names differ, use the existing replay batch/merge commands in `scripts/run_harness.py`. The required output is a replay JSON loadable by `load_replay_corpus()`.

## Step 5: Review with static dashboard

```bash
python scripts/review_dashboard.py \
  --input reports/replay/merged_v1_first50.json \
  --output-dir reports/review_dashboard/first50
```

Open:

```text
reports/review_dashboard/first50/index.html
```

Use the keyboard shortcuts:

- Right arrow: accepted / yes
- Left arrow: rejected / no
- `A`: expected abstain true
- `Z`: expected abstain false
- `S`: same entity
- `M`: moved entity
- `R`: renamed entity
- `N`: new entity same address
- `B`: branch ambiguity
- `O`: official current
- `T`: official stale
- `D`: official dead
- `G`: aggregator only
- `P`: parked domain
- `C`: official chain only
- `L`: official location page
- Space: skip
- Enter: next

Export `reviewed_replay.json` from the dashboard.

## Step 6: Run corpus report and gate

```bash
python scripts/replay_corpus_report.py \
  --input reports/review_dashboard/first50/reviewed_replay.json \
  --output reports/replay/corpus_v1_first50_report.json
```

For the first 25-50 cases, the v1 gate should fail. That is correct. Use the report to identify missing buckets.

To fail CI or a local script when the corpus is too thin:

```bash
python scripts/replay_corpus_report.py \
  --input reports/review_dashboard/first50/reviewed_replay.json \
  --fail-on-gate
```

Expected exit code for a thin first batch: `2`.

## Step 7: Fill missing buckets deliberately

Do not scale blindly. Read the report and fill what is missing.

Examples:

- not enough stale websites: sample more old-domain / official-stale cases
- not enough wrong branch cases: sample more chain homepage vs location page cases
- not enough aggregator echo: sample more Yelp/Facebook/directory-as-website cases
- not enough identity drift: sample moved/formerly/new-address cases
- not enough abstention cases: sample tied-authority and weak-evidence cases

Repeat:

```text
prioritized workplan -> evidence URLs -> fetch/enrich -> replay -> review -> corpus report
```

Stop only when the v1 corpus gate passes.

## Definition of shippable v1

The corpus tooling is shippable when:

- prioritized workplans can generate 25-50 high-value cases
- evidence pages enrich into replay-grade columns
- review dashboard can label cases quickly
- corpus report can explain why the corpus passes/fails
- reviewed replay JSON loads cleanly
- the v1 gate fails thin corpora and passes only when target shape is met

A passing corpus gate is necessary but not sufficient for resolver quality. After the corpus exists, resolver evaluation should add:

- accuracy on resolved cases
- coverage
- abstention rate
- correct abstention rate
- false abstention rate
- high-confidence wrong rate
- false official website rate
- stale official miss rate
- wrong branch error rate
- aggregator official confusion rate
